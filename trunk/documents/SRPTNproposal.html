Proposal for S, RP, TN Documents

Standards

Standards are required of all implementations. They therefore are the baseline, mandatory functionality.

From the DCC Working Group:

"An NMRA Standard is a public document that specifies the minimum requirements for interchange of equipment. To be granted a conformance seal, a manufacturer must meet all relevant requirements specified in all standards. "
S 9.6 - Base NMRAnet functionality
Includes startup procedures, etc. The base includes sending and receiving Events and Streams.
S 9.6.1 - Base NMRAnet functionality on CAN segments
All the messages and interactions of the base functionality, in CAN form; requires S 9.6
S 9.6.2 - Base NMRAnet functionality on TCP/IP links
All the messages and interactions of the base functionality, in TCP/IP form; requires S 9.6

Recommended Practices

Recommended Practices describe optional features and capabilities. Those need not be present, but if they are present they must function as described in the RP.

From the DCC Working Group:

"An NMRA Recommended Practice is a public document that specifies the maximum requirements for interchange of equipment. For any DCC product to be granted a conformance seal, the product must not violate any recommended practice. "
RP 9.6.1 – Some Optional Function in NMRAnet
Includes the messages and interactions for …
RP 9.6.1.1 - Some Optional Function on CAN segments
... as specifically handled on CAN segments
RP 9.6.1.1 - Some Optional Function on TCP/IP links
... as specifically handled on TCP/IP segments
RP 9.6.2 - Remote Configuration of NMRAnet Nodes
Using Events and Streams (and ??) to configure NMRAnet nodes remotely.
RP 9.6.3 - ??

Technical Notes

Technical Notes are non-normative documents which convey information useful to the model railroading community.

From the DCC Working Group:

"An NMRA Technical Note is Technical Department document used to convey information useful to the model railroading community. Technical Notes are only available to NMRA members."
DPH: This last sentence seems ominous – we will want these to be available to all NMRAnet users.
BobJ: I agree. Maybe the NMRA has some other type of doc we can use. Or we can write a “For Dummies” book outside the auspices of the NMRA...

There doesn't seem to be a single system of numbering for Technical Notes. Some are based on dates (TN-2004-1), some are linked to standards (TN-9.2.1), and some have multipart numbers TN-3-05. The following numbers are just notional.

TN 9.6.1 - Overview and Glossary
An introduction to the overall picture, justification, etc.
TN 9.6.2 - Assigning Unique Node and Event IDs
A discussion of how the various IDs are uniquely assigned. (Is this actually normative, and needs to be in the Standard or RP? Or does the S/RP just need to say "You must use an ID assigned to you by NMRA published procedures.")
TN 9.6.3 - NMRAnet Networking
Techniques for bridging, filtering and translating NMRAnet messages
TN 9.6.4 - NMRAnet for modular and ad-hoc layouts
Techniques for using NMRAnet on modular layouts that need to be rapidly reconfigured; for using NMRAnet when aggregating modular layouts into large exhibition layouts.
